PCI 101

  • PCI = Payment Card Industry
  • DSS = Data Security Standard
  • Holistic Approach
  • 12 requirements (aka the dirty dozen)
  • We will (quickly) go through them.

Dirty Dozen

  • 1. Install and Maintain a Firewall.
  • 2. Do Not Use Vendor Supplied Default Passwords.
  • 3. Protect Stored Data.
  • 4. Encrypt transmission of cardholder data across open, public networks.
  • 5. Use and regularly update anti-virus software or programs.
  • 6. Develop and maintain secure systems and applications.
  • 7. Restrict access to cardholder data by business need-to-know.
  • 8. Assign a unique ID to each person with computer access.
  • 9. Restrict physical access to cardholder data.
  • 10. Track and monitor all access to network resources and cardholder data.
  • 11. Regularly test security systems and processes.
  • 12. Maintain a policy that addresses information security for all personnel.

Not Optional

  • Accepting payment card transactions is a privelege, not a right.
  • PCI-DSS compliance is a contractual obligation.
  • Recommendation: treat PCI compliance as a minimum security policy.

So... How Do I Get Started?

  • Key Factors: Volume & Validation Type.
  • Volume determines PCI Level (1, 2, 3, or 4)
  • Validation type determines SAQ (A, A-EP, B, C, C-VT, D)
  • SAQ = Self Assessment Questionnaires
  • Provides checklist for 12 requirements.

Volume

Validation Type

Validation Type (English Please!)

  • SAQ A: Fully outsourced handling of sensitive data.
  • SAQ A-EP: "Mostly" outsourced handling of sensitive data.
  • SAC D: Storing sensitive data.

Merchant Managed

  • SAQ D (Danger)
  • Signficant liability on the merchant.
  • 327 security controls.
  • Unfortunately, MANY options out there for Drupal.
  • Basically, Drupal touches the card data.

Shared Management

  • SAQ A or A-EP
  • Signficant less risk.
  • 14-139 security controls.
  • Let's Review...

Direct Post

  • SAQ A-EP
  • Form Action or JavaScript
  • Tokenization

Hosted Payment Page

  • SAQ A
  • Redirect via HTML Form Action

iFrame

  • SAQ A
  • Essentially an Inline Hosted Payment Page

Shared-Management Vulnerabilities

  • Direct Post: Key Logger
  • Hosted Payment Page: Form Action Change
  • iFrame: Replace iFrame
